```typescript
import pLimit from 'p-limit';
import { loadFirms, loadRecipient } from './config/loader.js';
import { readState } from './state/reader.js';
import { fetchAll } from './pipeline/fetch.js';
import { dedupAll } from './pipeline/dedup.js';
import { summarize } from './summarize/gemini.js';
import { composeDigest } from './compose/digest.js';
import { sendMail } from './mailer/gmail.js';
import { writeState } from './state/writer.js';
import { scrubSecrets } from './util/logging.js';
import type { FirmResult, SummarizedItem } from './types.js';

async function main(): Promise<number> {
  try {
    const firms = await loadFirms();
    const recipient = await loadRecipient();
    const fromAddr = process.env.GMAIL_FROM_ADDRESS ?? recipient;
    const seen = await readState();

    const fetched = await fetchAll(firms);
    const deduped = dedupAll(fetched, seen);

    const summarizeLimit = pLimit(3);
    const summarized: FirmResult[] = await Promise.all(
      deduped.map(async (r) => {
        if (r.error || r.new.length === 0) return r;
        const out: SummarizedItem[] = await Promise.all(
          r.new.map((item) =>
            summarizeLimit(async (): Promise<SummarizedItem> => {
              // SUMM-06 / B3: NEVER pass title to Gemini. If description is absent, skip the call
              // entirely and return a placeholder SummarizedItem. Composer (plan 08) renders
              // summary_ko: null as "요약 실패 — 원문 확인".
              if (!item.description) {
                return {
                  ...item,
                  summary_ko: null,
                  summaryConfidence: 'low' as const,
                  summaryModel: 'skipped',
                };
              }
              return summarize(item, item.description);
            }),
          ),
        );
        return { ...r, summarized: out };
      }),
    );

    const newTotal = summarized.reduce((n, r) => n + r.summarized.length, 0);

    if (newTotal > 0) {
      const payload = composeDigest(summarized, recipient, fromAddr);
      await sendMail(payload);        // THROWS on failure — caught below, exit 1
    } else {
      console.log('No new items today — skipping email (DEDUP-03).');
    }

    await writeState(seen, summarized);   // OPS-03: state AFTER mail (also seeds bootstrap via r.raw per B1)
    return 0;
  } catch (err) {
    console.error('FATAL:', scrubSecrets((err as Error).message));
    return 1;
  }
}

main().then((code) => process.exit(code));
```

Execution-order invariants (OPS-03 + Pitfall 1):
1. loadFirms and loadRecipient happen FIRST — fail fast on bad config before any I/O.
2. seen = readState — happens BEFORE fetchAll (deduping needs seen).
3. fetched = fetchAll — each firm wrapped in try/catch by pipeline/fetch.ts; never throws.
4. deduped = dedupAll — pure, can't throw.
5. summarized = per-item summarize wrapped in pLimit(3). **B3**: description-missing items bypass Gemini via the `if (!item.description)` guard; `summarize()` itself (which never throws) handles the rest.
6. IF newTotal > 0: compose + sendMail — sendMail CAN throw; catch sets exit 1.
7. writeState — runs in both branches (newTotal > 0 AND newTotal === 0). In zero-new case with bootstrap, writer seeds urls from r.raw (B1). In zero-new non-bootstrap case, writer just updates lastUpdated.
8. return 0; top-level `.then(code => process.exit(code))` sets GHA exit status.

Critical failure mode: if sendMail throws and writeState had run BEFORE, a successful retry would read seen.json with the items already recorded → dedupAll returns new: [] → empty digest → silent item loss. This is Pitfall 1. The ordering above prevents that.

DEDUP-03 note: the check `if (newTotal > 0)` enforces "only send email when there's new content". Zero-new day = skip send, still update state (timestamp moves forward — keeps `lastUpdated` fresh for OPS-05 staleness detection in Phase 3).

**SUMM-06 / B3 note:** Plan 06 summarizer is told to receive a REAL article body in `body`. Passing `item.title` as a fallback body violates SUMM-06 (title was supposed to stay out of Gemini entirely) AND wastes quota on uninformative prompts. This file is the ONLY enforcement site — plan 06 `summarize()` trusts its caller. The `summaryModel: 'skipped'` marker distinguishes "Gemini intentionally not called" from `'failed'` (Gemini called and erred) in logs and state audits. Plan 08 composer renders `summary_ko: null` as "요약 실패 — 원문 확인" regardless of whether the confidence source was 'skipped' or 'failed' (plan 08 was revised separately — see W2 note there).

pLimit(3) for summarize (FETCH-03 spirit — cap parallel Gemini calls). Phase 1 has ~1-3 new items per day from Cooley, so this is scaffolding; Phase 2's multi-firm expansion benefits from the cap.

No retry at this level. Plan 06 (Gemini) and Plan 09 (Gmail) handle their own retries. Plan 11 is pure composition.
