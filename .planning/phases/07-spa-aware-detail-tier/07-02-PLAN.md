<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| YAML config → Playwright navigation | A firm's `item.url` is derived from scraped list-page HTML (hostile-to-friendly-ish), then passed to `page.goto()`. Phase 7 expands the set of URLs reaching Playwright (now includes bkl/kim-chang detail URLs). |
| External site → Playwright browser | Playwright loads a remote page in chromium and reads its DOM; a malicious page could attempt resource exhaustion or ETIMEDOUT. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-03 | Denial of Service | `page.goto(item.url)` hanging indefinitely → GHA run exceeds 2000min/mo budget | mitigate | `DETAIL_PAGE_TIMEOUT_MS = 15_000` is passed as `timeout:` to `page.goto()`. Playwright throws a TimeoutError after 15s. The outer per-item try/catch swallows it and returns the original item — sibling items continue. Total worst-case: 15s × N items × M firms; with N=10 items × 6 js-render firms = 900s = 15min, still under the 3-min-per-run practical budget because the 500ms INTER_FETCH_DELAY_MS gate keeps fetches serial within a firm but firms run in parallel via upstream pLimit(3). |
| T-07-04 | Denial of Service | `newContext()` / `newPage()` leak on partial failure → zombie contexts accumulate in long runs | mitigate | `finally { await ctx.close(); }` inside the per-item try block ensures context closes even on Playwright throw. The outer try/catch does NOT hold the context — only the inner try does. |
| T-07-05 | Information Disclosure | A malicious detail URL redirecting to a credential-grab page (SSRF-adjacent) | accept | All detail URLs originate from list-page scrape (public firm websites). No user-supplied URLs enter this path. Personal-tool threat model — no secrets in the browser context (USER_AGENT only); no session cookies carried across firms (per-item newContext). Browser runs headless `--only-shell` with no persisted storage. |
| T-07-06 | Tampering | A firm's HTML response intentionally serves JavaScript that modifies other firms' data | accept | Each firm gets a fresh `browser.newContext({ userAgent: USER_AGENT })` per item; cookie/storage isolation is per-context. No cross-firm data leak path. |
</threat_model>
