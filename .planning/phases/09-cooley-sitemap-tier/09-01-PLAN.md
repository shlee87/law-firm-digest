<decisions>
**Shape decision:** Extend the existing single-schema + superRefine pattern (not discriminatedUnion). Phase 4's locked precedent applies with equal or greater force at 4 variants. PATTERNS.md Pattern F.

**latest_n location:** Declared on the top-level FirmSchema `.object({...})` as `z.number().int().positive().optional()`. Enforced exclusivity via superRefine (only valid when type === 'sitemap'). Default value (10) is NOT baked into the schema — it belongs at the scraper layer (`DEFAULT_LATEST_N` constant in src/scrapers/sitemap.ts, plan 02). Rationale: schema default would shadow the `latest_n: 10` line in config/firms.yaml that D-06 says must be explicit and self-documenting; keeping the default at the scraper layer preserves the "explicit in YAML" intent even if a future migration accidentally drops the field.

**superRefine structure:** Replace the two-branch if/else ladder (js-render / else) with a three-branch ladder (js-render / sitemap / else-for-rss-html). This keeps the existing wait_for rule intact for js-render AND rss/html, adds sitemap-specific forbids (wait_for / selectors / detail_tier all invalid on sitemap), and gates latest_n exclusivity.

**detail_tier handling on sitemap firms:** Schema REJECTS explicit `detail_tier` on sitemap firms (path: ['detail_tier']). Sitemap tier implicitly routes through Playwright detail fetch — plan 03's enrichBody gate short-circuits on `firm.type === 'sitemap'`, so zod's `detail_tier='static'` default value never fires for sitemap firms. This is Pitfall 6's mitigation surface.
</decisions>

## Security Threat Model (ASVS L1)

**Scope:** Config parsing boundary. User-authored `config/firms.yaml` passes through `FirmsConfigSchema.parse()` at startup. This plan extends the schema surface; threats are around what the extended schema accepts or rejects.

| Threat | Severity | Mitigation |
|--------|---------|-----------|
| T1: Malicious YAML adds an unexpected `latest_n` on non-sitemap firm and silently overrides runtime behavior | LOW | superRefine rejects `latest_n` when type !== 'sitemap' with a custom issue (fail-fast at startup, not runtime) |
| T2: Malicious YAML declares type='sitemap' + detail_tier='static' to downgrade a sitemap firm to raw undici fetch (which CF-blocks Cooley) | MEDIUM | superRefine rejects `detail_tier` on any sitemap firm — the implicit-js-render contract cannot be overridden by YAML |
| T3: Malicious YAML supplies a `latest_n` of `Infinity`, `NaN`, or a negative integer to exhaust memory or crash sort | LOW | `z.number().int().positive()` rejects all three |
| T4: Prototype pollution via unexpected top-level fields | LOW | `.strict()` on FirmSchema rejects unknown keys (existing invariant — preserved) |
| T5: Downstream code reads `firm.latest_n` without null-check and crashes at runtime for non-sitemap firms | LOW | Optional `latest_n?: number` field — all downstream readers (scraper layer only) apply `firm.latest_n ?? DEFAULT_LATEST_N` |

**No HIGH severity threats.** This plan is a pure discriminator extension; no network I/O, no credentials, no user input beyond the already-validated config file.

**Residual risk:** None. All threats above are either trivially mitigated by existing validation primitives or architecturally impossible to reach.
</threat_model>
