    ```typescript
    export async function writeStepSummary(
      recorder: Recorder,
      firms: FirmConfig[],
      markers: ClusterMarker[] = [],
    ): Promise<void> {
      // D-12: no-op when the env var is unset (local runs, check:firm runs).
      const path = process.env.GITHUB_STEP_SUMMARY;
      if (!path) return;

      // Build payload: table always; markers section only when non-empty
      // (D-15: clean-run invisible posture). SINGLE appendFile call wraps
      // both writes so a half-success cannot leave the file inconsistent
      // (Pitfall 5).
      const table = recorder.toMarkdownTable(firms);
      let payload = table + '\n';
      if (markers.length > 0) {
        const lines = markers
          .map(
            (m) =>
              `- **${m.firmId}**: HALLUCINATION_CLUSTER_DETECTED — ${m.count} items demoted`,
          )
          .join('\n');
        payload += `\n## ⚠ Data Quality Warnings\n\n${lines}\n`;
      }

      try {
        await appendFile(path, payload, 'utf8');
      } catch (err) {
        // Phase 3 Pitfall 10: never propagate. Observability failure must not
        // surface as a red workflow. Scrub secrets defense-in-depth.
        console.warn(
          `[step-summary] write failed: ${scrubSecrets((err as Error).message)}`,
        );
      }
    }
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ClusterMarker.firmId / .count → step-summary markdown | firmId is config-controlled (safe by zod validation at startup); count is integer |
| ClusterMarker.signature → step-summary | NOT rendered in step-summary (signature is debug-only, stays in stderr marker per D-16) |
| appendFile payload → GITHUB_STEP_SUMMARY file | runtime-controlled path; GHA truncates at 1MB (our writes are <3KB) |
| ClusterMarker content → scrubSecrets | the D-15 section does NOT route marker text through scrubSecrets — firmId is a YAML-config string, safe by construction |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-06-01 | Tampering (markdown injection via firmId) | writeStepSummary | mitigate | firmId comes from config/firms.yaml, validated by zod at startup; it's a slug (`/^[a-z0-9][a-z0-9-]*$/`-shaped) — no markdown metachars. If a config change loosens this, the write could emit uncontrolled markdown. Task 1 markdown format uses `- **${m.firmId}**:` — a hostile firmId with `**` inside would close the bold. Mitigation: rely on zod firm-id validation (existing Phase 1 invariant). If further defense-in-depth needed, Plan 10 can add an escape helper; out-of-scope here. |
| T-08-06-02 | Tampering (log injection via scrubSecrets bypass in markers) | writeStepSummary markers payload | accept | Markers content is firmId + count — neither is user-controlled at runtime. The summary prefix is NOT in the step-summary output (it's debug-only in stderr per D-16). No scrubSecrets pipeline needed for this content. Matches CONTEXT.md §canonical_refs note: "scrubSecrets pipeline is this token... safe". |
| T-08-06-03 | Availability (half-success on partial write corrupting step-summary) | writeStepSummary | mitigate | Task 1 Pitfall-5 fix: table + markers section concatenated into single `payload` string, single `appendFile` call. If write fails, both pieces fail together (catch-block warns). Task 2 Pitfall-5 test asserts single call. |
| T-08-06-04 | Denial of Service (1MB GHA truncation on oversized markers list) | writeStepSummary | accept | Each marker line ~80 bytes; even 1000 clustered firms = 80KB, far below 1MB limit. Realistic upper bound: 12 firms × ~80 bytes = 1KB. Research §run-time-state confirms non-issue. |
</threat_model>
